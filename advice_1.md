## 1) 学生端 App 的产品边界

基于文档，你现在后端是“单管理员账号登录”，token 也暂时不做严格鉴权拦截。
学生端现实做法建议分两阶段：

* **阶段 1（竞赛/内测）**：学生端也用一个账号登录（或免登录），绑定 1 个设备（A-302-strip01），把体验做稳
* **阶段 2（长期）**：加“学生账号/绑定关系/权限”，否则学生端和管理员端概念会混淆（这一步可以后期做，不影响你现在 UI/交互）

## 2) 信息架构建议（4 Tab，学生端最实用）

1. **概览**：当前风险 + 快捷操作 + 最近曲线/事件
2. **插排**：实时插孔控制（主场景）
3. **告警**：离线/高功率/异常事件汇总（可处理）
4. **我的**：账号、设备绑定、通知、系统状态（可选 health）

> 你网页偏“中后台”，App 需要更“行动导向”：先把“风险、操作、反馈”放第一屏。

## 3) 页面与接口映射（严格对齐你现在已实现的接口）

### 3.1 登录页

* `POST /api/auth/login` 保存 token（哪怕当前不强校验，也要把链路留好）

### 3.2 概览页（Home）

首屏策略按你时序图来：先 REST 拉静态，再 WS 增量更新。

* 首屏：`GET /api/devices`（拿 online/offlineReason/lastSeen）
* 设备选中后：`GET /api/devices/{id}/status`（总功率、电压电流、sockets）
* 小曲线：`GET /api/telemetry?device={id}&range=60s`（做最近 60s/5min）
* 再连接 WS：`/ws`（DEVICE_STATUS / TELEMETRY / DEVICE_OFFLINE / CMD_ACK）

**Home 该展示什么（学生端最有效）：**

* 在线状态 + 离线原因（offlineReason 必须可见）
* 当前总功率（大数字）
* 快捷按钮：**一键断电**、**全部开启**（谨慎）、**仅保留插孔1**（“学习模式”）
* 最近事件（用 WS 推送拼出时间轴，或后端未来补 events API）

### 3.3 插排页（Device）

* `GET /api/devices/{id}/status` 初始化插孔矩阵
* 控制：`POST /api/strips/{device_id}/cmd`（socket/action 或 mode/policy）
* 命令结果确认：

  * 主通道：WS `CMD_ACK` 
  * 兜底：`GET /api/cmd/{cmdId}` 

**插孔卡片建议（移动端适配）：**

* 两列网格 SocketCard
* 每张卡必须有：开关（带 pending 动画）、当前功率、设备标签（device 字段）、状态 tag
* 顶部加“总开关 + 一键断电”悬浮操作条（便于单手）

### 3.4 告警页（Alerts）

你现在 WS 有 `DEVICE_OFFLINE`，并且设备列表会返回 `offlineReason`。
建议把“告警”定义为：

* 离线（offlineReason）
* 超功率/疑似违规（后续由 AI/规则产生）
* 命令失败/超时（CMD failed/timeout）

短期没有 alerts API 时，你可以用：

* WS 事件 + 本地列表缓存（最近 N 条）
* 或在插排页/概览页展示“最近异常”入口

### 3.5 我的（Me）

* 展示 token 对应用户信息（login 返回 user）
* 系统状态：`GET /health`（可放“服务正常/MQTT连接/DB连接”）
  这在工程长期可用里非常加分：学生端能区分“设备离线”还是“服务异常”。

## 4) 最关键：把“工程状态”做成 App 可理解的交互

你后端已有三件工程能力，App 必须把它们“显性化”，否则用户会觉得不稳定：

### 4.1 命令冲突 409（同目标 pending 时拒绝）

App 处理建议：

* 若返回 409：不要 toast 一句结束
* 直接在对应插孔卡上提示：**“已有指令执行中”** + 展示那个 cmd 的状态（调用 `GET /api/cmd/{cmdId}` 或后端未来可返回 pendingCmdId）
* 给“取消/重试”入口（取消可以先不实现，重试即可）

### 4.2 双通道确认（WS ACK + 轮询兜底）

建议统一命令体验：

* UI 进入 pending 后，先等 WS 1.5~2s
* 若没收到 ACK，开始每 500ms 轮询 `/api/cmd/{cmdId}`，最多 5s
* 5s 后标记 timeout，给“重试”按钮

### 4.3 离线机制（timeout + LWT 即时离线）

App 处理建议：

* 收到 `DEVICE_OFFLINE`：立刻全局置灰控制 + 顶部 banner 显示 reason
* 离线时点击开关：直接提示“设备离线，无法控制”，不要真的发 cmd（减少无效请求）
* `offlineReason` 在设备页、概览页都要能看到（这很工程）

## 5) 针对你文档里“deviceId 含空格需 URL 编码”的坑，App 设计要规避

你现在示例里 deviceId 有 `"A-303 strip01"`（带空格），会导致：

* 路由参数、缓存 key、埋点都很难受

**工程建议（强烈）：**

* 统一 deviceId 不要空格：用 `A-302-strip01`（你已采用）
* 后端对旧数据做兼容映射（可选）

App 侧无论如何都要做：

* `encodeURIComponent(deviceId)` 再拼 URL

## 6) UI 风格：从中后台 Web 到移动端“实用型”改造点

你 Web 是 Ant Design + ECharts，中后台风格较“信息密集”。App 建议：

* KPI 精简：只保留“在线/总功率/今日电量/告警数”
* 图表缩小：Home 只给 1 条迷你折线，点进才给 24h/7d
* 控制优先：插孔开关要大、要单手易点（Switch 旁加整行点击热区）
* 强反馈：pending/ack/timeout 都要可视（小圆点/进度条/短文本）

## 7) 技术实现建议（基于你现有栈）

你问的是 App，但你当前最省成本、最贴合现有技术栈的路线是：

### 方案 A：先做“移动端 H5/PWA”（推荐起步）

* 继续用 Next.js 15 + Antd + ECharts
* 做移动端布局与交互（TabBar、抽屉、卡片）
* 接入你现有 REST + WS，逻辑完全复用
* “添加到主屏幕”即可像 App 一样用

### 方案 B：再做 React Native（需要时）

当你需要：

* 真推送
* 蓝牙/本地权限
* 上架应用市场
  再转 RN。

> 你现在的后端与 WS 设计对 PWA/RN 都通用，先 PWA 不会走弯路。

## 8) 你下一步怎么做最有效（建议）

基于你的时序图，我建议你先把学生端 MVP 定为 3 个页面：

1. **Home**：设备状态卡 + 一键断电 + 60s 曲线 + 最近事件（WS）
2. **Device**：插孔矩阵 + cmd/ack（含 409/timeout）
3. **Alerts**：离线/命令失败/异常汇总（先由事件流生成）

这三页就能把“工程闭环”讲得非常漂亮。
